## LRN-20260628-002: Stop-Process -Name python 无差别杀进程

**时间**: 2026-06-28
**优先级**: CRITICAL
**状态**: VERIFIED
**领域**: 安全

### 教训
`Stop-Process -Name python -Force` 会杀掉所有 python.exe，
可能误杀网关进程。正确做法：按端口号精确定位。

```powershell
# 正确: 只杀特定端口
$pid = (Get-NetTCPConnection -LocalPort 10280).OwningProcess
if ($pid) { Stop-Process -Id $pid -Force }
```

## LRN-20260701-001: 多入口UI操作 — 只改一处遗漏另一处

**时间**: 2026-07-01
**优先级**: HIGH
**状态**: VERIFIED
**领域**: 前端

### 错误
修复2（切换设备页面刷新状态）时，我在 `switchPage()` 中加了 `refreshDevices()`。
但实际触发的是 `initNav` 的 click handler（第612行），那里仍然调用 `loadDevicesQuick()`。
`switchPage()` 是另一个入口，initNav 的 handler 根本不调用它。

### 根因
前端同一操作有多个入口（switchPage函数 + initNav点击handler + 按钮onchange）。
只找到并修改了一个入口，遗漏了另一个。

### 教训
``
修改UI行为 → 搜索所有触发路径 → 逐一确认修改
  - 按钮 onclick/onchange
  - 页面导航 handler（initNav）
  - 全局函数（switchPage）
  - 键盘快捷键
不要只改一处就认为完成
```

## LRN-20260701-002: 异步竞态覆盖 — 两个请求写同一个UI

**时间**: 2026-07-01
**优先级**: HIGH
**状态**: VERIFIED
**领域**: 前端/异步

### 错误
initNav 点击设备按钮同时触发：
```
loadDevicesQuick()  [GET, 无status, 异步]
switchPage → refreshDevices()  [POST, 有status, 异步]
```
refreshDevices 先返回 → 渲染"已连接" ✅
loadDevicesQuick 后返回 → 覆盖为 undefined → 渲染"离线" ❌

### 根因
两个异步请求都调用 renderDevices()，后返回的覆盖先返回的。
无status的GET覆盖了有status的POST结果。

### 教训
```
当两个异步操作更新同一个UI：
  方案A: 合并为一个请求（最佳）
  方案B: 用标志位阻止旧请求覆盖新数据
  方案C: 用 Promise.all 等两者完成后统一渲染
绝对禁止：两个独立async同时写同一个状态变量
```

## LRN-20260701-003: 多分支函数遗漏分支 — 只检查了一条路径

**时间**: 2026-07-01
**优先级**: HIGH
**状态**: VERIFIED
**领域**: 前端/逻辑完整性

### 错误
connectDevice() 有3个成功路径：
1. credentials获取成功 → 连接成功（主路径）
2. credentials API失败 → fallback无密码连接
3. credentials弹窗 → connectDeviceWithCreds()

路径2和3都有 `showToast('success', '设备已连接')`，但路径1只有 `addLog` 没有 `showToast`。
我只检查了路径2/3，认为"已有toast"，遗漏了路径1。

### 教训
```
检查函数完整性时：
  - 列出所有 if/else/catch 分支
  - 逐一检查每个分支是否都有用户反馈
  - 不要只看"有"的分支就认为全部有
```

## LRN-20260622-002: 理解目录结构的设计意图

**时间**: 2026-06-22 12:25
**优先级**: HIGH
**状态**: VERIFIED
**领域**: 架构设计

### 洞察

目录结构设计应该考虑：

1. **扩展性**：是否支持未来添加新内容？
2. **通用性**：是否适用于多种场景？
3. **层次性**：不同层级代表什么含义？

## LRN-20260622-003: 状态指示器应该反映实际状态

**时间**: 2026-06-22 12:25
**优先级**: HIGH
**状态**: VERIFIED
**领域**: UI设计

### 洞察

状态指示器/按钮应该：

1. **动态反映实际状态**：不要显示固定文本
2. **根据条件显示/隐藏**：已登录时不显示登录按钮
3. **提供准确反馈**：让用户知道系统当前状态

### 根因

在这次交互中，我错误地理解了状态指示器的作用：

❌ 错误做法：
```javascript
// 硬编码状态文本
status.textContent = "媒体输入: 等待连接"
```

✅ 正确做法：
```javascript
// 根据实际状态动态更新
if (isConnected && isStreaming) {
    status.textContent = "媒体输入: 播放中"
} else if (isConnected && !isStreaming) {
    status.textContent = "媒体输入: 已连接"
} else {
    status.textContent = "媒体输入: 等待连接"
}
```

### 实际应用

**场景1: 媒体输入状态**

应该显示：
- "等待连接" - 设备未连接
- "已连接" - 设备已连接但未播放
- "加载中..." - 正在加载视频流
- "播放中" - 视频正在播放
- "错误" - 播放出错

**场景2: 媒体输出状态**

应该显示：
- "未录制" - 没有录制任务
- "录制中" - 正在录制（绿色）
- "录制错误" - 录制出错（红色）
- "推流中" - 正在推流（未来功能）

**场景3: 登录按钮**

应该根据登录状态显示/隐藏：
- 未登录：显示"登录"按钮
- 已登录：不显示"登录"按钮，或显示"已登录"状态

### 检查清单

实现状态指示器时必须确认：

- [ ] 状态指示器应该显示哪些状态？
- [ ] 每种状态对应什么实际条件？
- [ ] 如何检测这些条件？
- [ ] 按钮/控件应该如何根据状态显示/隐藏？

## LRN-20260622-004: 区分不同信息框的职责

**时间**: 2026-06-22 12:25
**优先级**: HIGH
**状态**: VERIFIED
**领域**: UI设计

### 洞察

当界面有多个信息框/面板时，必须明确每个的职责：

1. **不要混淆职责**：每个信息框显示不同类型的内容
2. **信息分类**：关键事件 vs 详细日志 vs 操作记录
3. **用户期望**：用户期望在哪里看到什么信息？

### 根因

在这次交互中，我混淆了两个信息框的职责：

❌ 错误理解：
```
左侧媒体信息框 = 显示所有日志
右侧操作日志框 = 显示操作记录
```

✅ 正确理解：
```
左侧媒体信息框 = 只显示关键事件
  - 登录/登出
  - 码流切换
  - 截图操作
  - 录像开始/停止
  - WASM状态（初始化/加载成功/开始播放）
  - 格式：hh:mm:ss + 信息
  - 颜色区分：错误红色、一般灰色、成功绿色

右侧操作日志框 = 显示详细操作记录
  - 所有PTZ操作
  - 所有配置更改
  - 所有测试结果
  - 详细日志信息
```

### 实际应用

**设计多个信息框时**：

1. **问用户**：每个信息框应该显示什么？
2. **明确职责**：
   - 信息框A：关键事件（精简）
   - 信息框B：详细日志（完整）
   - 信息框C：操作记录（特定类型）
3. **格式化**：
   - 时间格式：hh:mm:ss
   - 颜色区分：成功/错误/一般
   - 图标区分：不同类型用不同图标

## LRN-20260704-001: 状态机设计要先规划完整状态图

**时间**: 2026-07-04
**优先级**: CRITICAL
**状态**: VERIFIED
**领域**: 算法设计

### 错误
对焦算法从v8.35开始多次补丁式修改，每次发现问题就打补丁，导致多轮迭代才完成重构。根因：没有先完整规划所有状态和转换条件，而是边做边改。

1. **先画状态图**：列出所有状态和转换条件
2. **定义每个状态的行为**
3. **按状态图写代码**，而不是边实现边调整逻辑

### 应用
ContrastAF类的状态机：
`
COARSE → PRECISE_RETURN → PRECISE_APPROACH → DONE/FAILED
     ↓          ↓                ↓
    反向      回退到best       缓慢逼近
`

### 检查清单

- [ ] 所有可能的状态是什么？

## LRN-20260704-002: 检测条件与拟合条件的点数要匹配

**时间**: 2026-07-04
**优先级**: HIGH
**状态**: VERIFIED
**领域**: 数学建模

### 错误
4点下降检测要求看4个点，但拟合要求best两侧各≥4点。当检测触发时，best是第1个点，右侧只有3个点，拟合永远失败。

### 修复
拟合条件改为右侧≥3点（因为4点下降已经保证右侧有3个有效点）。同时增加无效点过滤：连续点绝对值误差<10%视为无效。

### 检查清单

## LRN-20260704-005: 精调方向逻辑要基于反差变化，不基于固定规则

**时间**: 2026-07-04
**优先级**: CRITICAL
**状态**: VERIFIED
**领域**: 算法设计

### 错误
精调阶段使用"下降2步→反向"的固定规则，导致在峰值附近来回振荡221步无法收敛。0.05s步长太小，每次移动量太小，2步检测太慢。

### 修复
改为"反差 < best2_value → 立即反向"。1步即检测到越过峰值，立即反向继续逼近。

### 核心教训
```
精调方向决策应基于反差变化（实时信号），不基于固定步数规则
越过best（反差下降）= 立即反向
越接近best，步长应越小
```

### 检查清单
设计逼近算法时必须确认：
- [ ] 方向决策基于什么信号？
- [ ] 检测延迟是多少步？
- [ ] 在峰值附近是否会振荡？
- [ ] 是否有步数上限防止无限循环？

## LRN-20260704-006: best3与best2的关系

**时间**: 2026-07-04
**优先级**: HIGH
**状态**: VERIFIED
**领域**: 算法设计

### 定义
- best1 = 粗调阶段找到的峰值
- best2 = 精调起点（回退后的反差）
- best3 = 精调过程中的最高反差

### 要求
- best3 >= best2（精调必须比起点更好）
- best2 >= best1 才算成功（精调超越粗调峰值）

### 实现
```python
if contrast > self._best2_value:
    self._best2_value = contrast
if contrast > self._best3_value:
    self._best3_value = contrast
```

## LRN-20260704-007: 撞墙逻辑 — best1<100直接失败

**时间**: 2026-07-04
**优先级**: HIGH
**状态**: VERIFIED
**领域**: 算法设计

### 错误理解
我曾认为best1<100可以进入精调，在精调中继续寻找更高反差。

### 正确逻辑
- best1<100 → 直接失败，不进入精调
- 撞墙 = best1≥100 + 过best后下降 + 下降后稳定(10%波动)
- 进入精调的前提是best1≥100

### 核心教训
```
best1<100说明场景反差不足，精调也无法改善
不要在低反差区域浪费精调步数
```

**时间**: 2026-07-04
**优先级**: HIGH
**状态**: VERIFIED
**领域**: 设备控制

### 教训
对焦和白平衡移动后，设备需要时间稳定。如果不等待，截图时画面还在变化，反差值不准确。

await asyncio.to_thread(self._stop_focus)
await asyncio.sleep(0.3)  # 等待稳定
```

### 应用
- 对焦：每步移动后0.3s等待
