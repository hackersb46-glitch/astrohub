## 🔴 错误 #1: 主码流切换失败但没有重置播放标志

**现象**：
```
[04:35:50] 播放请求: stream=1 (主码流)
[04:35:50] 码流切换: 主码流
[04:35:48] 播放失败: 未知错误
[04:35:48] 播放调用失败: status=undefined
```

**根因**：
- `clickStartRealPlay` 的 error 回调中，`g_bPlaying2` 标志没有被重置
- 导致即使播放失败，代码仍认为正在播放

**影响**：
- 主码流无法切换成功
- 状态显示"视频播放中"但实际未播放

**修复位置**：`astrohub/src/web/index.html` `clickStartRealPlay` 函数

## 🟡 错误 #2: 第三码流直接播放成功但 channel 转换问题

**现象**：
```
[04:35:47] 播放请求: stream=3 (第三码流)
[04:35:47] 码流切换: 第三码流   ← changeStreamType 只在切换时触发，第一次播放不经过
[04:35:45] ● 视频播放中
```

**推测根因**：
- 第一次播放使用默认值 `g_iStreamType = 2` (子码流)
- `changeStreamType` 通过 cookie 设置 `webVideoCtrlProxyWsChannel=103`
- stream_wasm.py 解析：`ch_id = 103//100 = 1`, `st_type = 103%100 = 3`
- 但相机可能不支持 channel 1, stream 3，或者解析错误

**影响**：
- 第三码流播放异常（可能播放了错误的流）

**需要验证**：
- 相机是否支持 channel 1, stream 3 的组合
- **甚至可能不存在"第三码流"**，相机只支持主码流、子码流

## 🔴 错误 #3: channel 转换逻辑可能与相机实际格式不匹配

**当前逻辑**：
```javascript
// changeStreamType 中
var channelNum = 100 + parsedType;  // 1->101, 2->102, 3->103
document.cookie = 'webVideoCtrlProxyWsChannel=' + channelNum + '; path=/';
```

```python
# stream_wasm.py 中
ch_num = int(channel)  # 102, 103
ch_id = ch_num // 100  # 1, 1
st_type = ch_num % 100  # 2, 3
channel_path = f"/{ch_id}/{st_type}"  # /1/2, /1/3
```

**问题**：
- 是不是相机的 channel ID 不是从101开始？
- 是不是相机的码流类型映射不是除100取余？

**需要验证**：
- **必须观察相机返回的真实数据**
- 查看 `I_GetDigitalChannelInfo` 返回的实际 channel 列表
- 查看相机配置文件，确认码流类型定义

**证据**：
- 只有两个可选项：[主码流, 子码流, 第三码流]
- 但相机可能只支持前两种
