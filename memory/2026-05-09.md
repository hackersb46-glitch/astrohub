# 复盘记录 — 2026-05-09 20:52

## 核心错误模式

### 错误 1：任务定义偏差（形式 vs 实质）
- **表现**：让 OMO "核对代码有没有调用端点"，不是"按 CSV 方法实现功能"
- **后果**：代码调用了端点，但参数错、返回值处理错、状态流转错
- **修改**：任务描述必须 = "读 CSV 列 D 方法 → 实现对应功能 → 写测试验证列 E 标准"

### 错误 2：形式审查代替实质验证
- **表现**：16 commits，16 个 ✅，0 个实际运行验证
- **后果**：PTZ 摇杆无反应、限位不动、预置点为空、速度写死
- **修改**：功能完成标志 ≠ commit，= 实际运行通过测试

### 错误 3：跳过测试环节
- **表现**：`py_compile` 语法检查当作"测试通过"
- **后果**：所有 ISAPI 端点调用都没用真实设备验证
- **修改**：每个方法必须对应有自动化测试脚本

### 错误 4：质量把关缺失
- **表现**：信任 OMO 自报完成，不启动服务、不操作验证
- **后果**：左中右三栏丢失没发现，OSD 报错没发现
- **修改**：每次 OMO 完成后必须实际运行验证核心路径

## 正确流程（从此执行）

```
读 CSV → 逐条实现（列 D 方法） → 逐条测试（列 E 标准） → 实际运行验证 → 打勾
```

每步的验证标准：
- ✅ 代码实现正确（py_compile + import）
- ✅ 单元测试通过（模拟 API 调用，比对返回值）
- ✅ 端到端通过（真实设备/摄像头，确认物理响应）
- ✅ UI 符合 SPEC（截图对比）

## 关键教训
- CSV 是标准答案，严格按 CSV 写 = 结果正确
- "核对" ≠ "实现"，"语法正确" ≠ "功能可用"
- 我（开发工厂）的核心职责是质量把关，不是催 OMO 产出
- 宁可进度慢，不可功能不可用
